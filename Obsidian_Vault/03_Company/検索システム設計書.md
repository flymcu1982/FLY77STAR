# 社内検索システム 設計書

## 目的

新しく入った人間やAI社員が、分からないことを人に聞かなくても**5分以内**に理解できる会社にする。

- Obsidianを会社の知識ベースにする
- GitHubを正式なデータソースにする
- 検索だけで、関連ノート・Knowledge・Projects・People・Promptへ辿り着けること
- 将来的にAI社員も同じ検索システムを使う

[[00_Start_Here/FLY77STAR_Philosophy|Philosophy]] の「探さない」原則を、具体的な仕組みに落とし込むための設計書です。

## 基本方針: 新しい検索エンジンは作らない

まず結論から書きます。今の規模(このVaultとGitHubリポジトリ)では、専用の検索エンジンを新たに構築する必要はありません。Obsidianの検索・エイリアス機能と、GitHubのコード検索が、すでに十分な検索エンジンです。

設計すべきは「エンジン」ではなく、**検索した時に正しい場所へ一発で着地し、そこから関連情報へ芋づる式に辿れる構造**です。

## 3層構成

### 層1: エンティティ・ハブノート(すぐ導入できる)

- 会社に存在する「名前を持つもの」(プロジェクト、キャラクター、ツール、概念、ルール用語)は、それぞれ**1つだけ正式なノート(ハブノート)**を持つ。同じ名前を複数箇所に書き散らさない。
- ハブノートのfrontmatterに `aliases`(別名・英語表記・略称)を持たせる。Obsidianの検索・クイックスイッチャーは、どの呼び方で検索してもそのノートに一発で着地する。
- 軽い用語(1行で説明できるもの。例: Dreamina, Higgsfield)は [[00_Start_Here/用語集|用語集]] のエントリー自体がハブノート。内容が育つもの(プロジェクト、キャラクターなど)は専用ファイルを持つ。
- すべてのハブノートは、末尾に固定フォーマットの「関連」セクションを持つ。

```
## 関連
- Knowledge: [[...]]
- Projects: [[...]]
- People: [[...]]
- Characters: [[...]]
- AI Employees: [[...]]
- Prompt: [[...]]
- GitHub(正本): `path/to/file`
```

これが「検索すると関連ノート・Knowledge・Projects・People・Promptが表示される」を実現する具体的な仕組みです。ソフトウェアではなく、**ノートの書き方の約束事**で実現します。

People(実在の人間・スタッフ)、Characters(作品内キャラクター)、AI Employees(AI社員・AI部署)は、それぞれ `04_Team/People/`・`10_Characters/`・`04_Team/AI_Employees/` と、はっきり分けています。検索した時に人物・キャラクター・AI社員が混ざらないようにするためです。

### 層2: 索引ノート(すぐ導入できる)

- `00_Start_Here/索引.md` に、存在する全ハブノートを1行1件、リンクと一言説明で列挙する。
- 目的は2つ。(1) 検索が苦手な人がブラウズできる保険。(2) AI社員がVault全体を素早く把握するための、単一のエントリーポイント(1ファイル読むだけで「会社に何が存在するか」の一覧が手に入る)。
- 完璧な同期は目指さない。多少漏れても実害は小さい(検索自体はaliasesで機能するため)。Knowledge Reviewや年次棚卸しのタイミングで追記していく運用で十分。

### 層3: 横断インデックス生成スクリプト(将来、必要になったら)

- Vaultが育ち、層1・層2の手作業が追いつかなくなったら、`scripts/` にある既存の自動化スクリプト群と同じ思想で、Obsidian VaultとGitHubリポジトリの両方をスキャンし、「用語 → 存在する場所(Obsidianノート/GitHubファイルパス)」の対応表を自動生成するPythonスクリプト(stdlib限定)を追加できる。
- **今は作らない。** 層1・2を実際に運用してみて、手作業が破綻してから着手する([[00_Start_Here/FLY77STAR_Philosophy|Philosophy]]の「必要になってから作る」の実践)。

## GitHubとの関係

検索の入口は常にObsidian側(ハブノート)。GitHub側の内容は複製せず、ハブノートの「GitHub(正本)」欄でファイルパスにリンクするだけ。GitHub上のルール文面そのものを検索したい場合は、GitHubのコード検索を使う。

Obsidianのエイリアス検索と、GitHubのコード検索、この2つの検索面をはっきり分け、どちらも「本文の二重管理」をしないことで、10年後もズレが生じない設計にする。

## 3年後・10年後に壊れやすい点への対策

1. **名前の衝突**: 10年運用すると、同じ名前が複数の意味で使われることが起きる。実際、導入初日に「CUT」で衝突が起きた(制作用語のCUT01〜CUT18と、AI社員候補の「CUT」)。→ ハブノートのタイトルは曖昧さがあれば `CUT (AI Employee)` のように曖昧さ回避の接尾辞を付け、`aliases` に短い呼び方(`CUT`)を残す。両方存在する場合はObsidianが両方を検索結果に出すので、迷わず選べる。将来キャラクター名が衝突した場合も同じ手法(例: `KAI (SEVENTHS)`)で対応する。
2. **エンティティの種類が増える**: 今はProjects/People/Knowledge/Promptだが、10年後には「楽曲」「ブランド」など新しい種類が増えているはず。→ ハブノートは「どのフォルダに置くか」ではなく「関連セクションのフォーマット」で成立する仕組みなので、新しい種類が増えてもフォルダ構成を作り直す必要がない。
3. **索引ノートの陳腐化**: 手作業のインデックスは追加漏れが起きる。→ 層3の自動生成スクリプトへ移行する判断基準は、「索引ノートに載っていないハブノートが増えてきたと感じたら」。
4. **GitHubファイルパスのリンク切れ**: ファイルが将来リネーム・移動される可能性がある。→ ディレクトリ構成自体は `FOLDER_STRUCTURE.md` で安定運用されているためリスクは低いが、年次棚卸しのタイミングでリンク切れも一緒に確認する。

## AI社員が使う場合の注意

現時点で実際にこのVaultを「読んで検索できる」AI社員は、会話型のAI(Claude Code、ChatGPT/Claude(チャット))に限られます。Higgsfield・Seedance・Palmier・ElevenLabsのような生成専用ツールは、自分でVaultを検索する仕組みを持っていません。これらのツールに情報を渡す際は、人間か会話型AIが該当ノートを探して渡す運用のままになります。「AI社員全員が自律的に検索する」という将来像に近づけるには、別途そのための連携が必要で、これは今回の設計の範囲外です。
